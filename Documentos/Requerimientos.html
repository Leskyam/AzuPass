Documento de requerimientos para proyecto Web - AzuPass

SUSTENTO DE ESTE DOCUMENTO:
    Este documento pertenece a la fase conceptual del proceso de diseño del proyecto antes mencionado, basándose en el estilo de trabajo de Microsoft Solutions Framework (MSF). Durante esta fase, el equipo de diseño identifica los conceptos y objetivos principales de la aplicación desde el punto de vista del usuario y la empresa. Al final del proceso, el equipo habrá desarrollado el documento de especificaciones de funcionamiento de la aplicación que constituirán la base para cualquier tarea de diseño posterior.
AUDIENCIA:
Objetivo(s): Determinar qué usuarios o clientes harán uso de la solución
    La audiencia identificada para este servicio Web XML podrá estar compuesta por cualquier aplicación que lo requiera y que sea capaz de consumir servicios de este tipo (XML Web Services), entonces tenemos que podrá ser utilizado, al menos, por aplicaciones y otros servicios desarrollados utilizando cualquier entorno de desarrollo .Net, tanto aquellas que se ejecutan sobre HTTP como las de escritorio. Estas aplicaciones o servicios que consumirán de éste (AzuPass) a partir de ahora serán llamados indistintamente "clientes" ó "aplicaciones clientes".
DESCRIPCIÓN GENERAL DE LA FUNCIONALIDAD:
Objetivo(s): Exponer de manera generalizada los requisitos con los que debe cumplir el producto final.
    AzuPass es un servicio Web XML que manipulará registros de usuarios en una base de datos personalizada con el fin de proporcionar a sus aplicaciones clientes un mencanismo centralizado para la identificación de los mismos, para ello expondrá los métodos que se entiendan necesarios a través de los cuales las aplicaciones clientes podrán implementar funciones básicas como: identificar usuarios, recuperar datos de perfiles de usuarios, recuperar lista de las aplicaciones que hacen uso de este servicio y otras que se consideren propias de este tipo de servicio. Además del propio servicio existirá una aplicación cliente "Administrador de perfiles AzuPass" que permitirá hacer todas las funciones necesarias para el manejo de los perfiles desde el punto de vista del usuario final, lo cual implica que los desarrolladores de aplicaciones que empleen este servicio para identificación de usuarios no necesitan implementar los mecanismos propios para la manipulación de los perfiles del usuario en la aplicación que desarrollen, sino que podrán remitir a los usuarios a la URL de la aplicación que lo hace (Administrador de perfiles AzuPass), implementando solamente en sus aplicaciones lo necesario para la valildación de las credenciales que proporciona el usuario y cualquier otro método que se considere pertinente y que le esté permitido a las aplicaciones clientes.
CASOS DE USO:
Objetivo(s): los casos de uso son simplemente descripciones textuales de la interacción que tiene lugar entre un agente externo llamado Actor1 y la solución que se está diseñando. Los actores también pueden ser servicios, componentes y usuarios, entre otros. La información más importante que se debe determinar para la creación de los casos de uso es la acción del actor y el resultado esperado.
    Usuario: aplicación cliente (Cualquier Aplicación que pueda consumir este servicio Web XML)
    1. Valida credenciales de usuario (_LogOn)
    2. Valida credenciales de usuario y devuelve datos de éste (_LogOnPlus)
    3. Devuelve las aplicaciones favoritas (más utilizadas) de un usuario (_getProfileFavorites)
    4. Devuelve las aplicaciones que hacen uso de este servicio Web XML (_getServicedApplications)
    5. Devuelve las direcciones de correo-e de los administradores de este servicio (_getServiceAdmins)
    6. Obtener la URL de la aplicación que gestiona los datos de los perfiles de usuario (_getProfileManagerURL)
    7. Obtener código HTML con el mensaje "Acerca de" de este servicio (_getAbout)
    8. Obtener la URL de la imagen de logo de este servicio (_getLogoImageSrc)
    9. Obtener el nombre corto de este servicio (_getServiceShortName)
    10. Obtener el nombre largo de este servicio (_getServiceFullName)
    11. Obtener lista de categorías ocupacionales (_getListCatOcupacionales)

    Notas.

    • Todos los escenarios que aquí se describen tienen en cuenta que se está argumentando la funcionalidad de un componente o servicio, omitiendo las especificaciones de la implementación que se haga por parte del desarrollador de la aplicación cliente, las descripciones se hacen tomando en cuenta los valores que se necesitan para cada método y los posibles valores devueltos en cada caso. En el caso de errores se devolverán tales como son siempre que sea posible o una descripción correspondiente para evitar proporcionar información sencible que pueda ser utilizada para comprometer la seguridad o el funcionamiento del servicio.
    • Cada método que se exponga para ser utilizado por la aplicación cliente mostrará una descripción detallada del mismo y de cada uno de sus parámetros tanto de entrada como de salida, siempre y cuando se emplee un entorno integrado de desarrollo que cuente con esta capacidad.

    Usuario: aplicación cliente (Aplicaciones que pueden consumir este servicios)
    1. Crear nuevo registro del usuario (addUserProfile)
      Para crear un nuevo registro de usuario se requiere que la aplicación cliente proporcione los siguientes datos: Nombre, apellidos, dirección de correo electrónico, año de nacimiento, mes de nacimiento, día de nacimiento, sexo, categoría ocupacional y nombre de la aplicación a través de la que se registra. El servicio tendrá en cuenta y registrará los siguientes datos: fecha y hora del registro. Se validará la siguiente información: el dominio de correo debe estar dentro de "*.minaz.cu", la fecha que se forma producto de la unión del año, mes y día de nacimiento debe ser válida. En caso de ser efectivo el registro del usuario el servicio enviará la contraseña (un valor encriptado autogenerado) a la dirección de correo proporcinada por el usuario final a través de la aplicación cliente. En caso de no proceder se devolverá un objeto del tipo error "exception" que contendrá la causa que lo ha impedido.

    2. Validar credenciales del usuario (valUserPass)
      Para validar las credenciales del usuario se requiere que la aplicación cliente proporcione los siguientes datos: nombre de la aplicación cliente que hace uso del servicio, dirección de correo electrónico del usuario y su contraseña. El servicio tendrá en cuenta y registrará los siguientes datos: fecha y hora de esta validación de las credenciales y el nombre de la aplicación que se proporciona. En caso de ser efectiva la verificación de las credenciales proporcionadas por el usuario final el servicio devolverá un valor del tipo verdadero "true", de lo contrario devolverá falso "false". En caso de error "exception" se devolverá la descripción del error.

    3. Cambiar contraseña del usuario (chgUserPassword)
      Para cambiar la contraseña del usuario se requiere que la aplicación cliente proporcione los siguientes datos: dirección de correo electrónico del usuario, contraseña en uso actualmente y la nueva contraseña. El servicio validará que la contraseña cumpla con los requerimientos que se exponen a través de la cadena devuelta por el método "passwordRequeriments", el cual se podrá utilizar para mostrar al usuario esta información. En caso de ser efectiva la verificación de las credenciales en uso proporcionadas por el usuario final se hará el cambio de la contraseña y se devolverá un valor del tipo verdadero "true", de lo contrario devolverá falso "false". En caso error "exception" se devolverá la descripción del error.

    4. Cambiar datos del perfil del usuario (chgUserProfile)
      Para cambiar los datos del perfil del usuario se requiere que la aplicación cliente proporcione las credenciales del mismo. El servicio validará las credenciales y en caso de ser efectiva la verificación de las credenciales se aplicarán los cambios solicitados, si el procedimiento es efectivo se devolverá un valor del tipo verdadero "true", de lo contrario devolverá falso "false". En caso error "exception" se devolverá la descripción del error.

    5. Obtener datos del perfil del usuario (getUserProfile)
      Para obtener los datos del perfil del usuario se requiere que la aplicación cliente proporcione las credenciales del usuario en cuestión y en caso de ser efectiva la verificación de las credenciales proporcionadas se devolverán los los mismos datos que para el registro inicial con excepción de la contraseña, de lo contrario se devolverá error "exception" con la correspondiente descripción.

    6. Obtener URL del logo del servicio (getLogoURL)
      Para obtener el logo del servicio no es preciso proporcionar ningún parámetro. Se devolverá una cadena con la URL de la ubicación de la imagen que representa el logo de este servicio. Esta imagen se puede utilizar cerca de los controles donde el usuario proporcionará las credenciales para llamar su atención e indicarle que está haciendo uso del servicio "AzuPass" para su identificación.

    7. Obtener requerimientos de la contraseña (getPasswordRequeriments)
      Para obtener los requerimientos de la contraseña no es preciso proporcionar ningún parámetro. Se devolverá una cadena con la descripción de las características de la contraseña esperada. Esta descripción podrá ser utilizada cerca del control de nueva contraseña para llamar su atención e indicarle que debe cumplir con los requerimientos de la misma.

    8. Obtener URL de la aplicación "Administrador de perfiles AzuPass" (getProfileMgrURL)
      Para obtener la URL de la aplicación que manipula los perfiles del usuario no es preciso proporcionar ningún parámetro. Se devolverá una cadena con la dirección URL donde se hospeda dicha aplicación. Esta dirección podrá ser mostrada como un hypervínculo para remitir a los usuarios que requieran hacer cambios en su perfil a la misma y evitar así la implementación en su aplicación de otros métodos además del de la validación de las credenciales del usuario.

    9. Obtener trazas de usuarios (getUserLog)
      Cada usuario podrá ver sus propios registros de actividad, en ellos se mostrarán las aplicaciones que ha utilizado y las fechas y horas a las que lo ha hecho. Además habrán algunos usuarios declarados como "SuperAdmins" a nivel de este servicio, estos usuarios podrán utilizar la aplicación "Administrador de perfiles AzuPass" para ver los registros de actividad de todos los usuarios. Para utilizar este método deberán proporcionarse las credenciales del usuario.

    10. Eliminar registro de usuario (delUserProfile)
      Para eliminar un registro de usuario es preciso propporcionar las credenciales del usuario que se eliminará, el servicio valida las mismas y si están correctas, entonces eliminará cualquier registro del usuario. De realizarse el procedimiento de manera correcta se devolverá verdadero "true" de lo contrario se devolverá falso "false". De suceder algún error "exception" se devolverá una descripción según como corresponda.

    11. Obtener lista de categorías ocupacionales (lstCatOcupacionales)
      Para obtener las direcciones de correo electrónico que se configuren para soporte técnico a este servicio no es preciso proporcionar ningún dato de entrada. Estas direcciones pueden utilizarse para enviar correo con dudas o problemas que se presenten con la funcionalidad del servicio.

    12. Obtener lista de aplicaciones más utilizadas (lstMostUsedApps)
      Se podrá obtener una lista de las aplicaciones cliente más utilizadas por el usuario luego de estar éste identificado, para ello será necesario proporcionar los siguientes datos: dirección de correo del usuario, su contraseña y la cantidad de aplicaciones que se devolverá en la lista. Esta lista podrá ser utilizada para hacer vínculos a las direcciones URL de las aplicaciones más utilizadas por el usuario en cuestión y enriquecer la navegabilidad de su aplicación, esto es muy conveniente para aquellos que implementan Inicio de Sesión Unico (Single Sign On) en sus sitios Web o conjunto de aplicaciones.

    13. Agregar elementos a la lista de categorías ocupacionales (addCatOcupacional)
      Para agregar elementos a la lista de categorías ocupacionales es preciso que el usuario que está haciendo uso de la aplicación cliente del servicio se identifique como Administrador de este, enotonces la aplicación deberá proporcionar el nombre de la nueva categoría ocupacional que se necesita agregar. De ser efectiva la operación se envierá de retorno un valor verdadero "true", de lo contrario y en caso de suceder un error se devolverá el mismo o la descripción correspondiente.

    14. Eliminar elementos de la lista de categorías ocupacionales (delCatOcupacional)
      Para eliminar un elemento de la lista de categorías ocupacionales es preciso que el usuario que está haciendo uso de la aplicación cliente del servicio se identifique como Administrador de este, enotonces la aplicación deberá proporcionar el ID de la categoría ocupacional que se necesita eliminar y la operación se llevará a cabo si este no está en uso por alguno de los perfiles de usuario activo. De ser efectiva la operación se envierá de retorno un valor verdadero "true", de lo contrario y en caso de suceder un error se devolverá el mismo o la descripción correspondiente.

    15. Modificar elemenetos de la lista de categorías ocupacionales (chgCatOcupacional)
      Para modificar un elemento de la lista de categorías ocupacionales es preciso que el usuario que está haciendo uso de la aplicación cliente del servicio se identifique como Administrador de este, enotonces la aplicación deberá proporcionar el nuevo nombre de la categoría ocupacional que se necesita modificar. De ser efectiva la operación se envierá de retorno un valor verdadero "true", de lo contrario y en caso de suceder un error se devolverá el mismo o la descripción correspondiente.

    Nota.
    Los escenarios de uso y los casos de uso obtenidos a partir de éstos deben describir el comportamiento esperado de la aplicación en varios procesos empresariales. Además, debido a que el proceso de diseño es iterativo, se pueden incorporar nuevos casos de uso en el diseño si durante el proceso de desarrollo se identifica un nuevo tipo de interacción.
